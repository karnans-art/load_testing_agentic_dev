5. User Provisioning

Both projects create test doctor accounts via the Atlas Admin API. The approaches differ significantly:

StepProject A (Karnan)Project B (Suresh)
1. AuthenticationCDS_ADMIN_COOKIE env var (tlas cookie)ATLAS_SESSION env var (different cookie name)
2. Create userPOST /api/hubdoctorPOST /api/hubdoctor
3. Set passwordIMAP: fetch reset email, extract token, set passwordAssumes default password works (no verification)
4. Run testAutomatic (same command)Manual (separate command)
5. CleanupAuto-deactivate created usersNo cleanup
Automation levelFully automated, single commandManual, multi-step

Note: On the UAT environment, newly created users cannot login without the IMAP password reset flow. This was verified by creating a test user and attempting login — it returns 401 "Incorrect credentials". The IMAP approach is required for our environment.

6. Session Management

The CDS server allows only one active session per user. A second login invalidates the first token. This is the most critical design constraint for multi-VU testing.

AspectProject A (Karnan)Project B (Suresh)
VU-to-user mappingUSERS[__VU - 1] — direct index, skip if out of boundsUSERS[(__VU - 1) % length] — modulo wrap
Collision riskNonePresent — wraps when simulator VUs shift IDs
Single-user testsEach VU gets unique userAll VUs share one token from setup()
Token refreshOn 401, re-loginOn 401, re-login

Bug found in Project B: The modulo mapping (__VU - 1) % USERS.length causes two VUs to share the same user when the simulator scenario allocates VU IDs that push doctor VU IDs beyond the user count. This was identified and fixed in Project A.
