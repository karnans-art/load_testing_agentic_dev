1. Platform Overview

This platform simulates production traffic across multiple products using real data from WAF logs and HAR files. Each product (app) is tested independently under its own apps/ directory. Results reveal bottlenecks across the full infrastructure.

Currently active: CDS (Clinical Decision Support) — the ECG doctor workflow.

Designed for: 10-15 products, each with its own test suite, sharing common infrastructure (runner, Grafana, CI/CD).

FeatureDetails
Enginek6 (HTTP, WebSocket, Browser)
Monitoringk6 Cloud + Grafana/InfluxDB (local)
CI/CDGitHub Actions
User provisioningAutomated: create, test, deactivate
ScenariosSmoke, production, peak, stress, soak, websocket, diagnosis loop
Custom metricsPer-phase timing: dashboard load, ECG viewer, WebSocket, diagnosis

2. Quick Start

Prerequisites

node --version   # v20+
k6 version       # v1.7+
docker --version # for Grafana (optional)

Install

git clone <repo>
cd agentic-loadtesting
npm install
cp .env.example .env   # fill in credentials

Run your first test

# Health check (1 VU, ~5 seconds)
npm run health:cds

# Smoke test (5 VUs, 30 seconds)
npm run smoke:cds

# Smoke test with dynamic users
npm run smoke:cds -- --setup 4

# Push results to k6 Cloud
npm run smoke:cds -- --cloud

3. Architecture

agentic-loadtesting/
  apps/
    cds/                         # CDS product (reference implementation)
      scripts/
        smoke.js                 # Smoke test
        peak-load.js             # Peak (3x production)
        stress-test.js           # Stress (1x-5x ramp)
        soak-test.js             # Endurance (2 hours)
        production-load.js       # WAF-weighted production rate
        health-check.js          # Fast CI gate
        websocket-test.js        # Socket.IO load test
        full-doctor-flow.js      # Probabilistic routing
        continuous-diagnosis.js  # Diagnosis loop
        lib/
          cds-auth.js            # Per-VU auth (1:1 mapping)
          cds-workflow.js         # Doctor workflow (batch calls)
          cds-metrics.js          # Custom k6 metrics
          cds-websocket.js        # Socket.IO handler
      users.json                 # Hardcoded test users (DO NOT MODIFY)
      fixtures/sample.csi        # ECG sample file for simulator
    admin/                       # Future: admin product
    customer/                    # Future: customer product
  runner/
    run-test.js                  # Orchestrator (--setup, --cloud)
    setup/provision-lib.js       # User creation + IMAP + cleanup
  grafana/                       # Dashboard + InfluxDB config
  docker-compose.yml             # Grafana + InfluxDB stack
  .github/workflows/             # CI/CD pipeline
  .env                           # Environment config (gitignored)

App isolation rule: Each app under apps/ is completely self-contained. No imports across apps. Shared infrastructure lives in runner/ only.

Test execution flow

npm run smoke:cds -- --setup 4 --cloud

  1. Provision 4 users (API + IMAP password reset)
  2. Write temp users.json
  3. k6 runs smoke test (4 doctor VUs + 1 simulator VU)
     - Simulator pushes ECG cases
     - Doctors: login → list page → assign → diagnosis page
  4. Deactivate dynamically created users
  5. Restore original users.json
  6. Results pushed to k6 Cloud

5. User Provisioning

Static users (default)

Without --setup, tests use the hardcoded apps/cds/users.json. These are permanent test accounts that must not be modified or deleted.

Dynamic users

npm run smoke:cds -- --setup 10 --prefix ltkr

This creates 10 users (ltkr001–ltkr010), runs the test, and deactivates them after.

StepWhat happens
1. CreatePOST /api/hubdoctor via Atlas Admin API (CDS_ADMIN_COOKIE)
2. PasswordIMAP: batch fetch reset emails (single connection), extract tokens, set passwords
3. Testk6 runs with dynamic users (temp users.json)
4. CleanupPUT /api/hub/toggle-status deactivates only dynamically created users
5. RestoreOriginal users.json restored from backup

Important: Only users created in the current --setup run are deactivated. Hardcoded users (Karnan, User3, pooja.cs, lt001, lt002) are never touched.

Session collision prevention

The CDS server allows only one active session per user. Each VU gets a unique user via direct index mapping (USERS[__VU - 1]) with no modulo. If a VU's index exceeds the user count, it's skipped.

7. CI/CD Integration

Current setup (GitHub Actions)

On push to main (when apps/cds/** changes) or manual dispatch:

Checkout → Node.js 20 → npm install → Install k6 → Run smoke test → Push to k6 Cloud

Required GitHub secrets

SecretPurpose
BASE_URLTarget environment URL
K6_CLOUD_TOKENk6 Cloud API token
CDS_ADMIN_COOKIEAtlas Admin tlas cookie (for --setup)
TRICOG_ADMIN_COOKIESimulator admin cookie
GOOGLE_EMAILGmail for IMAP password resets
GOOGLE_APP_PASSWORDGmail app password

Post-deployment flow (planned)

CDS Deploy → Smoke test (30s gate) → Pass? → Production load (10m) → Report to k6 Cloud
                                       → Fail? → Alert team, block promotion

8. Adding a New Product

# 1. Create the directory structure
mkdir -p apps/<name>/scripts/lib apps/<name>/fixtures

# 2. Create the shared libs (copy from CDS, adapt)
#    - apps/<name>/scripts/lib/<name>-auth.js
#    - apps/<name>/scripts/lib/<name>-workflow.js
#    - apps/<name>/scripts/lib/<name>-metrics.js

# 3. Create test scripts
#    - apps/<name>/scripts/smoke.js
#    - apps/<name>/scripts/health-check.js
#    - (more as needed)

# 4. Create users.json
#    - apps/<name>/users.json

# 5. Add npm scripts to package.json
#    "smoke:<name>": "node runner/run-test.js --app <name> --script smoke"

# 6. Run
npm run smoke:<name>

Convention: npm scripts follow the pattern {scenario}:{app}. Examples: smoke:cds, stress:admin, health:customer.

10. Troubleshooting

ProblemCauseFix
401 UNAUTHENTICATED on multiple endpointsSession collision — two VUs share same userCheck VU-to-user mapping. Ensure no modulo in user index. Add more users if needed.
400 "Domain missing the username" on task/assignUsername not in task's assignable listMust call task/users?role=undefined first and use username from response.
CDS_ADMIN_COOKIE errorCookie expired or not setGet fresh tlas cookie from browser at uat-ecg-atlasadmin.tricogdev.net. Update .env.
IMAP timeout / no emailGmail rate limit or email delayRetry. Check GOOGLE_APP_PASSWORD is valid. Reduce --setup count.
Simulator push failsTRICOG_ADMIN_COOKIE expiredGet fresh cookie from uat-admin.tricogdev.net. Update .env.
Skipped users not usableUsers exist but password unknownUse a new --prefix to create fresh users. Or delete old users first.
Grafana shows no dataInfluxDB not receiving k6 outputRun with --out influxdb=http://localhost:8086/k6. Check docker compose is running.
